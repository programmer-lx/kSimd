# 核心开发规则

## 修改策略与流程
架构变更预审： 在涉及核心架构调整前，必须先在对话中说明方案，获得人工确认后方可修改文件。

存量逻辑保护： 严禁在无正当理由的情况下删除或修改现有逻辑。

新增逻辑限制： 严禁引入与当前任务无关的新逻辑。

## C++ 编程规范
C++标准： 严格使用 C++20 特性，优先使用 concepts 和 requires 子句。

文档： 所有复杂的模板元编程 (Template Metaprogramming) 代码必须包含详细的注释。

模式对齐： 必须严格遵循 @kSimd/kSimd/core/impl/ops 目录下的既有代码模式。

## 目录权限与边界
Agent 工作区： 仅限修改 @kSimd/kSimd/core/impl/ops 目录下的文件。

禁止修改区： 以下分发框架文件由人工维护，严禁主动修改（除非用户明确指示）：

base.hpp / base.cpp / dispatch.hpp / dispatch_core.hpp / dispatch_this_file.hpp

分发假设限制： 在编写 ops 实现时，必须视上述文件为既定的“黑盒”框架，不得假设可以同步修改分发系统。

## 指令集分发规则 (ISA Dispatch)
核心原则： KSIMD_DYN_DISPATCH_LEVEL 宏代表当前多重包含 (Multi-include) 展开中正在编译的具体指令集级别。

能力匹配： 仅允许使用当前 KSIMD_DYN_DISPATCH_LEVEL 明确授权的 Intrinsics。

禁止越权： 严禁在低级别路径调用高级别指令（例如在 V3 路径用 AVX-512）。

保留回退： 若当前级别不支持某高级指令，必须完整保留现有的低级别回退 (Fallback) 实现。

禁止随便升级指令： 不得因为 CPU “可能支持”而直接使用超出当前宏定义的指令。

### 宏语义说明
#### 边界宏 (*_START / *_END)
仅用于家族范围检查或包含控制，严禁将其作为功能判定的逻辑依据。

#### 级别宏 (X86_V1-V4, NEON, SVE 等)
这是功能判断的唯一依据。

X86 平台： V1 < V2 < V3 < V4 是严格递增的。

ARM 平台： 指令集是组件化的，必须使用 == 或 != 进行精确匹配，禁用 >= 判定。

## 特殊类型处理规则
FP16 逻辑
FULL_FP16 定义： 仅当宏包含 *_FULL_FP16 时，才允许使用原生 FP16 算术指令。

兼容路径： 在非 FULL_FP16 路径下，必须保留“提升至 f32 -> 运算 -> 转回 f16”的模拟路径。load_*函数提升类型，store_*函数转回fp16

## 函数声明和实现
必须统一使用 KSIMD_API(...) 模式，该模式自动集成了 KSIMD_DYN_FUNC_ATTR、强内联及调用约定。

禁止： 手动编写 __attribute__((target(...))) 或自行定义分发属性宏。

## 必须请示人工的情况
在出现以下任一情况时，必须停止操作并请求人工确认：

需要添加新的分发级别 (Dispatch Level)。

需要修改分发框架文件（如 base.hpp 等）。

发现 ops 目录下的实现与外部分发框架存在语义不一致。

建议删除任何现有的回退 (Fallback) 路径。
